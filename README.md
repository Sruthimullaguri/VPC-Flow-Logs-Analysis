# VPC Flow Logs Analysis with Athena using Terraform

## Project Overview

In this project I created one AWS VPC environment using Terraform and enabled VPC Flow Logs to monitor traffic between instances inside public and private subnets. After that I stored logs in S3 and analyzed them using Amazon Athena queries.

Main purpose of this project was to understand:

* how traffic flows inside VPC
* how security groups allow or block traffic
* how rejected connections appear in logs
* how DevOps engineers troubleshoot network issues using Flow Logs
* how Athena helps analyze infrastructure traffic without creating extra servers

This project helped me understand AWS networking observability in practical way.

---

## Infrastructure Setup Using Terraform

First I configured AWS provider inside **provider.tf** and used variable-based region configuration instead of hardcoding region values. I also configured default resource tags so all infrastructure created by Terraform automatically gets consistent tagging.

Then I created **variables.tf** where I defined reusable inputs like:

* aws_region
* vpc_cidr
* public_subnet_cidr
* private_subnet_cidr
* instance_type
* flowlogs_bucket_name

After that I created VPC inside **vpc.tf** and enabled DNS hostname resolution so instances inside VPC can communicate correctly.

Next I created **networking.tf** where I configured:

* public subnet
* private subnet
* internet gateway
* route table
* subnet route association

Public subnet is used for bastion instance and private subnet is used for internal instance testing.

Then I created **security.tf** where I configured security groups:

* SSH access allowed to bastion instance
* SSH allowed to private instance only from bastion security group
* ICMP allowed inside VPC for ping testing

After security configuration I created EC2 instances inside **compute.tf**

Instances created:

* bastion instance in public subnet
* private instance in private subnet

These instances helped generate traffic for Flow Logs testing.

Then I created S3 bucket inside **storage.tf** which stores Flow Logs generated by AWS.

After that I enabled Flow Logs inside **flowlogs.tf**

Configuration included:

* traffic type ALL
* capture ACCEPT traffic
* capture REJECT traffic
* aggregation interval configured

Then I created Athena database inside **athena.tf** to query Flow Logs stored in S3.

Finally I created **outputs.tf** to display important values after Terraform deployment like:

* bastion public IP
* private instance IP
* bucket name
* VPC ID

## Traffic Testing I Performed After Deployment

After infrastructure deployment I connected to bastion instance using SSH and generated different traffic types to verify Flow Logs capture correctly.

Command used:

ssh -i ~/sruthiai.pem ec2-user@bastion_public_ip

After login I generated internal traffic:

ping private_instance_ip

This generated ICMP traffic and also REJECT entries because security group blocked response.

Next I generated outbound internet traffic:

wget google.com

This generated TCP outbound traffic visible in Flow Logs.

# Then I generated blocked port connection traffic:

nc private_instance_ip 443

This generated REJECT traffic entries inside Flow Logs.

These traffic tests helped confirm Flow Logs captured both accepted and rejected traffic correctly.

## Issues I Faced During Implementation and How I Fixed Them

During implementation I faced multiple real-world issues.

First issue was SSH login failure after Terraform deployment. Later I found key pair was not attached in compute configuration. After adding correct key_name attribute inside compute.tf and running terraform apply again issue resolved.

Second issue happened because PEM file permission error inside WSL environment. SSH requires strict permission control. I moved PEM file into Linux home directory and applied chmod 400 permission. After that SSH connection worked.

Third issue happened when ping between bastion and private instance failed. Initially I thought configuration mistake exists but later I realized security group rules blocked ICMP response. This helped me confirm Flow Logs captured REJECT entries correctly.

Fourth issue happened when Athena Run button was disabled. Reason was Athena query output location not configured. After setting S3 output location queries executed successfully.

Fifth issue happened when Athena table creation failed due to Parquet format mismatch. Flow Logs were stored in log.gz format. I recreated table using correct text schema and issue resolved.

Sixth issue happened due to reserved keyword column name “end” inside Athena schema. I renamed column to end_time and recreated table successfully.

## Athena Queries I Executed for Traffic Analysis

After Flow Logs table creation I executed multiple SQL queries to analyze captured traffic.

First I checked top traffic source IP addresses using query:

SELECT srcaddr,
SUM(bytes) AS total_bytes
FROM vpc_flow_logs
GROUP BY srcaddr
ORDER BY total_bytes DESC
LIMIT 10;

This helped identify instances generating maximum traffic inside VPC.

---

Next I verified rejected traffic using query:

SELECT *
FROM vpc_flow_logs
WHERE action='REJECT';

This helped confirm blocked connections generated during ping and nc testing.

---

Then I verified SSH connection traffic using query:

SELECT *
FROM vpc_flow_logs
WHERE dstport=22;

This confirmed SSH connectivity between bastion and private instance.

---

Next I analyzed protocol-level traffic using query:

SELECT protocol,
COUNT(*) AS connection_count
FROM vpc_flow_logs
GROUP BY protocol;

Protocol values observed:

1 = ICMP traffic from ping test

6 = TCP traffic from SSH and wget commands

---

Then I verified communication between instances using query:

SELECT *
FROM vpc_flow_logs
WHERE srcaddr='10.0.1.xxx'
AND dstaddr='10.0.2.xxx';

This confirmed subnet-to-subnet internal traffic inside VPC.
